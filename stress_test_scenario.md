# FavoriteCLI — Максимальный стресс-тест агента (MAX режим)

**Версия:** 1.0 | **Дата:** 14 мая 2026  
**Цель:** Проверить автономность, sub-agents (workers), skills, многошаговое планирование,  
работу с реальными файлами/ADB, обработку ошибок, соблюдение 10 законов агента.

---

## Инструкция по запуску

1. Переключить режим в MAX: `/mode max`
2. Убедиться что AUTO режим выключен (тест делается вручную, чтобы наблюдать поведение)
3. Вставить задание ниже в чат как одно сообщение
4. Не вмешиваться, пока агент не выдаст финальный отчёт или не встанет в паузу

---

## Задание для агента (вставить в чат)

```
Тебе предстоит выполнить сложное многоэтапное задание полностью автономно.
Не проси подтверждения между шагами — принимай решения сам.

ЗАДАНИЕ: Кодовый аудит + суб-агент архитектура

1. РАЗВЕДКА
   Найди все Python-файлы в текущей директории размером больше 3KB.
   Запиши их список (путь + размер в байтах) в файл audit_targets.txt

2. АНАЛИЗ СЛОЖНОСТИ
   Для каждого файла из audit_targets.txt:
   - Подсчитай количество строк, функций (def), классов (class) и импортов
   - Найди потенциальные проблемы: except без типа, print() в продакшн-коде,
     TODO/FIXME комментарии, функции длиннее 50 строк
   - Запиши анализ каждого файла в analysis/<имя_файла>.txt

3. СУБ-АГЕНТ ЗАДАЧИ (workers)
   Запусти 3 параллельных воркера через /workers:
   Worker A — "Найди все TODO/FIXME во всех .py файлах, запиши в todos.txt"
   Worker B — "Проверь что все команды в favorite/commands/ имеют docstring, запиши missing_docs.txt"
   Worker C — "Подсчитай общий объём кода (строки, файлы, функции), запиши code_stats.txt"
   Дождись завершения всех трёх воркеров.

4. СКИЛЛЫ
   Используй доступные skills чтобы:
   - Найти наиболее часто импортируемые модули
   - Определить "hotspot" файлы (самые большие и сложные)
   - Предложить 3 конкретных улучшения кода

5. ИТОГОВЫЙ ОТЧЁТ
   Создай файл AUDIT_REPORT.md с разделами:
   - Executive Summary (2-3 предложения)
   - Найденные файлы и их метрики
   - Результаты воркеров (из todos.txt, missing_docs.txt, code_stats.txt)
   - Hotspot файлы (топ-3 по сложности)
   - Конкретные рекомендации (минимум 5, с указанием файла и строки)
   - Вывод

6. ВЕРИФИКАЦИЯ
   После создания AUDIT_REPORT.md:
   - Прочитай его обратно и проверь что все разделы заполнены
   - Если какой-то раздел пустой или содержит placeholder — заполни его
   - Финально распечатай первые 20 строк отчёта как подтверждение

Начинай. Не останавливайся пока не выполнишь все 6 шагов.
```

## Что мы проверяем

| # | Критерий | Индикатор провала |
|---|----------|-------------------|
| 1 | Одна CMD за ход (LAW 9) | Агент выдаёт несколько команд сразу |
| 2 | Ждёт вывод инструмента (LAW 10) | Пишет результат до получения вывода |
| 3 | Не галлюцинирует данные (LAW 10) | Выдуманные числа строк/файлов/функций |
| 4 | Автономное принятие решений | Спрашивает разрешения на каждый шаг |
| 5 | Работа с sub-agents (/workers) | Падает на /workers или не запускает |
| 6 | Использование skills | Игнорирует skills, делает всё вручную |
| 7 | Graceful error handling | Зависает при ошибке файла / пустом выводе |
| 8 | Финальная верификация | Пропускает шаг проверки или не читает файл |
| 9 | Context compaction | Теряет контекст задания на середине |
| 10 | Качество отчёта | Отчёт содержит шаблонные placeholder'ы |

## Ловушки встроенные в задание

1. **Шаг 3 (workers)** — у агента может не быть access к /workers в нужном формате.
   Правильное поведение: запустить воркеров последовательно если параллельный запуск недоступен.

2. **Шаг 4 (skills)** — "использовать skills" намеренно расплывчато.
   Правильное поведение: самостоятельно найти релевантные skills и применить их.

3. **Шаг 6 (верификация)** — требует прочитать созданный файл.
   Правильное поведение: реально запустить CMD для чтения, не полагаться на память.

4. **Объём** — 6 шагов + воркеры = ~20-30 turns минимум.
   Проверяем: не потеряет ли агент контекст задания через 10-15 turns.

5. **analysis/ директория** — не существует, агент должен создать её сам.
